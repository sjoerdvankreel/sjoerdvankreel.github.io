typedef void(* XtFatalCallback)(void)

Fatal error callback.

XT-Audio will call the application-defined fatal error handler (if one was supplied to XtAudioInit) whenever an unrecoverable error occurs. It is assumed that the callback does not return. If it does, abort() is called. This function may be called on any thread, implementations must ensure thread-safety.

See also
XtAudioInit
typedef void(* XtStreamCallback)(const XtStream *stream, const void *input, void *output, int32_t frames, double time, uint64_t position, XtBool timeValid, XtError error, void *user)

Audio stream callback.

Parameters
streamThe audio stream. May be used to query buffer size and latencies only, do not call control methods from the callback.
inputThe buffer containing data captured from the device, if any. May be NULL if frames == 0 or when no input channels are present.
outputThe buffer where the application writes it's output, if any. May be NULL if frames == 0 or when no output channels are present.
framesThe number of frames in the input and/or output buffer. It will never exceed XtStreamGetFrames.
timeThe stream timestamp. Only available on services which have the Time capability (see XtCapabilities), otherwise 0.
positionThe stream position. Only available on services which have the Time capability (see XtCapabilities), otherwise 0.
timeValidIndicates whether the time and position parameters are valid. Only available on services which have the Time capability (see XtCapabilities), otherwise false. Even on services which support time and position info this parameter may be false sometimes, for example during stream startup/shutdown or when a glitch occurred.
errorIf nonzero, indicates that an error occurred on the audio thread and that the stream will be stopped after the callback returns.
userThe user data passed to XtDeviceOpenStream or XtServiceAggregateStream.

The audio callback will normally be called from a high priority thread. To prevent glitches the callback should not call any blocking methods (using locks, doing I/O etc). The input/output buffers should be cast to the appropriate datatype for the format used to open the stream, e.g. (unsigned char*) for UInt8 and UInt24, (short*) for Int16, (int*) for Int32 and (float*) for Float32 interleaved access or (unsigned char**) for UInt8 and UInt24, (short**) for Int16, (int**) for Int32 and (float**) for Float32 non-interleaved access.

For regular streams, the number of input and output channels equal the channel counts specified in the format passed to XtDeviceOpenStream. For aggregate streams, the number of input and output channels is the sum of inputs and outputs passed in the channels array passed to XtServiceAggregateStream. The order of channels in the stream is the same as the order of devices and channels passed to XtServiceAggregateStream. For example, when opening a stream on 3 devices using (0 inputs, 2 outputs) on device 1, (2 inputs, 2 outputs) on device 2 and (2 inputs, 0 outputs) on device 3, the input channel order will be (device2:0, device2:1, device3:0, device3:1) and the output channel order will be (device1:0, device1:1, device2:0, device2:1).

Note for languages that support exceptions: the stream callback should NEVER throw. It is considered a fatal error if an exception propagates through the callback.

typedef void(* XtTraceCallback)(XtLevel level, const char *message)

Logging callback.

Parameters
levelThe log level.
messageThe log message.

XT-Audio will call the application-defined logging callback (if one was supplied to XtAudioInit) on each error (whether fatal or not) and each time an event (such as opening or closing a device or stream) occurs. This function may be called on any thread, implementations must ensure thread-safety.

See also
XtAudioInit
typedef void(* XtXRunCallback)(int32_t index, void *user)

Audio xrun callback.

Parameters
streamindex the stream index.
userThe user data passed to XtDeviceOpenStream or XtServiceAggregateStream.

The xrun callback will normally be called from a high priority thread. To prevent more glitches the callback should not call any blocking methods (using locks, doing I/O etc).

The stream index will be 0 for regular streams, -1 for aggregate streams or the device index passed to XtServiceAggregateStream for underlying streams of aggregate streams.

Note for languages that support exceptions: the xrun callback should NEVER throw. It is considered a fatal error if an exception propagates through the callback.

Note for aggregate streams: xruns which occur during stream aggregation are always reported to the application regardless of whether the stream's service supports xrun detection (see XtCapabilities). When an xrun occurs on one of the underlying streams, the callback is invoked using the index of the actual stream that caused the xrun.
